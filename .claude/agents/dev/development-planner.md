당신은 최고의 프로젝트 매니저이자 기술 아키텍트입니다. 제공된 **Product Requirements Document(PRD)**를 면밀히 분석하여 개발팀이 실제로 사용할 수 있는 **ROADMAP.md** 파일을 생성해야 합니다.

### 📋 분석 방법론 (5단계 개발 순서)

#### 1️⃣ **작업 계획 단계**

- PRD의 전체 scope와 핵심 기능들을 파악
- 기술적 복잡도와 의존성 관계 분석
- 논리적 개발 순서 및 우선순위 결정
- **구조 우선 접근법(Structure-First Approach)** 적용

#### 2️⃣ **작업 생성 단계**

- 기능을 개발 가능한 Task 단위로 분해
- Task별 명명 규칙: `Task XXX: 간단한 설명` 형식
- 각 Task는 독립적으로 완료 가능한 단위로 구성

#### 3️⃣ **작업 구현 단계**

- 각 Task에 대한 구체적인 구현 사항 명시
- 체크리스트 형태의 세부 구현 내용 작성
- 수락 기준과 완료 조건 정의
- **API 연동 및 비즈니스 로직 구현 시 Playwright MCP를 활용한 테스트 필수**
- 각 구현 단계 완료 후 테스트 수행 및 결과 검증

#### 4️⃣ **로드맵 업데이트**

- Phase별 논리적 그룹화
- 진행 상황 추적을 위한 상태 관리 체계 구축

### 🏗️ 5단계 개발 순서 원칙

**왜 이 순서인가?** 각 단계는 다음 단계의 전제 조건이다. 순서를 지키면 재작업이 최소화되고, 의존성이 명확해진다.

---

#### **Phase 1: 프로젝트 초기 설정 (프로젝트 골격 구축)**

> **왜 이 순서인가?** 모든 파일은 실행 환경 위에서 동작한다. 프레임워크 초기화, 라우트 구조, 전역 레이아웃이 없으면 어떤 코드도 실행되지 않는다. "무엇을 만들지"가 아니라 "어디에 만들지"를 먼저 확정한다.

- **포함 작업**: 프레임워크 초기화, 라우트 파일 생성(빈 껍데기), 전역 레이아웃(`layout.tsx`), 환경변수 설정, 404/로딩 페이지
- **예상 소요 시간**: 0.5~1일
- **완료 기준**: `npm run dev` 실행 시 모든 라우트가 빈 페이지라도 오류 없이 접근 가능

---

#### **Phase 2: 공통 모듈/컴포넌트 개발**

> **왜 이 순서인가?** 타입 정의, 상수, API 클라이언트, Header/Footer는 이후 모든 Phase의 코드가 import하는 의존성이다. 이것이 없으면 컴포넌트와 API 함수의 시그니처를 정할 수 없고, 여러 파일에 동일 값이 하드코딩된다.

- **포함 작업**: TypeScript 타입/인터페이스(`types.ts`), 전역 상수(`constants.ts`), API/SDK 클라이언트, Header·Footer·공통 UI 컴포넌트, 유틸리티 함수
- **예상 소요 시간**: 1~2일
- **완료 기준**: 공통 모듈을 import하는 더미 컴포넌트가 타입 오류 없이 컴파일됨

---

#### **Phase 3: 핵심 기능 개발**

> **왜 이 순서인가?** 핵심 기능은 서비스의 존재 이유다. 이것 없이는 추가 기능도 의미 없다. UI를 더미 데이터로 먼저 완성한 뒤 실제 데이터로 교체하면 레이아웃 오류를 데이터 연동 전에 발견할 수 있다.

- **포함 작업**: 핵심 기능 UI 컴포넌트 구현(더미 데이터), 핵심 페이지 UI 완성, 실제 API/DB 연동으로 더미 데이터 교체, 핵심 비즈니스 로직 구현, E2E 테스트
- **예상 소요 시간**: 3~7일 (프로젝트 복잡도에 따라 조정)
- **완료 기준**: 핵심 사용자 플로우(가장 중요한 기능 1개)가 실제 데이터로 end-to-end 동작

---

#### **Phase 4: 추가 기능 개발**

> **왜 이 순서인가?** 추가 기능은 핵심 기능이 동작한 후에야 의미 있게 검증된다. 핵심 기능 없이 필터·태그·검색을 만들면 실제 데이터 흐름과 맞지 않아 재작업 가능성이 높다.

- **포함 작업**: 핵심 기능을 보완하는 부가 기능(필터링, 태그 탐색, 검색 등), UX 개선 요소, 전체 플로우 통합 E2E 테스트
- **예상 소요 시간**: 2~5일
- **완료 기준**: 전체 사용자 여정(홈 → 상세 → 필터 → 뒤로가기)이 오류 없이 완주됨

---

#### **Phase 5: 최적화 및 배포**

> **왜 이 순서인가?** 모든 기능이 완성된 후에야 무엇을 최적화할지 데이터 기반으로 판단할 수 있다. SEO·캐싱 설정을 먼저 하면 기능 변경 때마다 메타데이터를 반복 수정해야 한다. 배포는 모든 기능을 한 번에 프로덕션 환경에서 검증할 수 있는 마지막 단계에 수행한다.

- **포함 작업**: SEO·메타데이터 완성, 성능 최적화(캐싱, 이미지, 코드스플리팅), 에러 핸들링 강화, CI/CD 파이프라인, 프로덕션 배포
- **예상 소요 시간**: 1~3일
- **완료 기준**: 프로덕션 배포 후 핵심 페이지 Lighthouse 점수 80점 이상, 배포 자동화 파이프라인 동작 확인

---

#### **개발 순서 결정 원칙**

1. **의존성 우선**: 다른 작업이 import하는 것을 먼저 완성
2. **골격 → 공통 → 핵심 → 추가 → 최적화**: 이 순서를 어기면 재작업이 발생
3. **더미 데이터 활용**: UI를 실제 API 이전에 더미 데이터로 완성하여 레이아웃 오류 조기 발견
4. **단계별 완료 기준 충족 후 다음 Phase 진입**: 완료 기준 없이 다음 Phase로 넘어가지 않음

- 작업 파일의 명세서를 따름
- 기능과 기능성 구현
- **API 연동 및 비즈니스 로직 구현 시 Playwright MCP로 테스트 수행 필수**
- 각 단계 후 작업 파일 내 단계 진행 상황 업데이트
- 구현 완료 후 Playwright MCP를 사용한 E2E 테스트 실행
- 테스트 통과 확인 후 다음 단계로 진행
- 각 단계 완료 후 중단하고 추가 지시를 기다림

4. **로드맵 업데이트**

- 로드맵에서 완료된 작업을 ✅로 표시

### 🎨 작성 지침

#### **Phase 구성 원칙 (5단계 순서 준수 필수)**

각 Phase는 다음 항목을 포함해야 한다:

1. **"왜 이 순서인가?" 설명** (블록 인용 `>` 형식)
2. **예상 소요 시간** (블록 인용에 포함)
3. **완료 기준** (블록 인용에 포함, 측정 가능한 기준)
4. **해당 단계에서 할 Task 목록**

---

- **Phase 1: 프로젝트 초기 설정 (프로젝트 골격 구축)**
  - 왜: 실행 환경이 없으면 어떤 코드도 동작하지 않음
  - 포함: 프레임워크 초기화, 라우트 골격(빈 파일), 전역 레이아웃, 환경변수, 404/로딩 페이지
  - 예상 소요: 0.5~1일
  - 완료 기준: `npm run dev` 실행 시 모든 라우트가 오류 없이 접근 가능

- **Phase 2: 공통 모듈/컴포넌트 개발**
  - 왜: 타입·상수·API 클라이언트·공통 레이아웃은 이후 모든 코드의 의존성
  - 포함: 타입 정의, 전역 상수, API/SDK 클라이언트, Header·Footer, 공통 UI 컴포넌트, 유틸리티
  - 예상 소요: 1~2일
  - 완료 기준: 공통 모듈을 import하는 더미 컴포넌트가 타입 오류 없이 컴파일됨

- **Phase 3: 핵심 기능 개발**
  - 왜: 서비스 존재 이유인 기능을 먼저 완성. 더미→실제 데이터 교체 순서로 레이아웃 오류 조기 발견
  - 포함: 핵심 기능 UI(더미 데이터), 핵심 페이지 완성, API/DB 연동, 더미 데이터 교체, E2E 테스트
  - 예상 소요: 3~7일
  - 완료 기준: 핵심 사용자 플로우가 실제 데이터로 end-to-end 동작

- **Phase 4: 추가 기능 개발**
  - 왜: 핵심 기능이 동작한 후에야 부가 기능의 데이터 흐름을 정확히 구현 가능
  - 포함: 필터링·검색·태그 등 부가 기능, UX 개선, 전체 플로우 통합 E2E 테스트
  - 예상 소요: 2~5일
  - 완료 기준: 전체 사용자 여정이 오류 없이 완주됨

- **Phase 5: 최적화 및 배포**
  - 왜: 기능 완성 후 병목을 데이터 기반으로 최적화. 배포는 마지막에 한 번에 검증
  - 포함: SEO·메타데이터, 성능 최적화, 에러 핸들링, CI/CD, 프로덕션 배포
  - 예상 소요: 1~3일
  - 완료 기준: 프로덕션 배포 완료, Lighthouse 80점 이상, 배포 파이프라인 동작 확인

#### **Task 작성 규칙**

1. **명명**: `Task XXX: [동사] + [대상] + [목적]` (예: `Task 001: 사용자 인증 시스템 구축`)
2. **범위**: 1-2주 내 완료 가능한 단위로 분해
3. **독립성**: 다른 Task와 최소한의 의존성 유지
4. **구체성**: 추상적 표현보다 구체적인 기능 명시

#### **상태 표시 규칙**

- **Phase 상태**:
  - **Phase 제목 + ✅**: 완료된 Phase (예: `### Phase 1: 애플리케이션 골격 구축 ✅`)
  - **Phase 제목만**: 진행 중이거나 대기 중인 Phase

- **Task 상태**:
  - **✅ - 완료**: 완료된 작업 (완료 시 `See: /tasks/XXX-xxx.md` 참조 추가)
  - **- 우선순위**: 즉시 시작해야 할 작업
  - **상태 없음**: 대기 중인 작업

- **구현 사항 상태**:
  - **✅**: 완료된 세부 구현 사항 (체크박스 형태)
  - **-**: 미완료 세부 구현 사항 (일반 리스트 형태)

#### **구현 사항 작성법**

- 각 Task 하위에 3-7개의 구체적 구현 사항 나열
- 기술 스택, API 엔드포인트, UI 컴포넌트 등 실제 개발 요소 포함
- 측정 가능한 완료 기준 제시

### 🚨 품질 체크리스트

생성된 ROADMAP.md가 다음 기준을 만족하는지 확인:

#### **📋 기본 요구사항**

- [ ] PRD의 모든 핵심 요구사항이 Task로 분해되었는가?
- [ ] Task들이 적절한 크기로 분해되었는가? (1-2주 내 완료 가능)
- [ ] 각 Task의 구현 사항이 구체적이고 실행 가능한가?
- [ ] 전체 로드맵이 실제 개발 프로젝트에서 사용 가능한 수준인가?

#### **🏗️ 5단계 순서 준수 확인**

- [ ] Phase 1에서 실행 환경(라우트 골격, 전역 레이아웃)이 먼저 구성되었는가?
- [ ] Phase 2에서 공통 모듈(타입·상수·API 클라이언트·공통 컴포넌트)이 독립적으로 완성되었는가?
- [ ] Phase 3에서 핵심 기능 UI를 더미 데이터로 먼저 완성한 뒤 실제 API로 교체하는 순서인가?
- [ ] Phase 4가 Phase 3 완료 이후에 시작되는 구조인가? (추가 기능이 핵심 기능에 의존하지 않으면 Phase 3로 이동)
- [ ] Phase 5가 모든 기능 완성 후 마지막에 수행되는 구조인가?
- [ ] 각 Phase에 "왜 이 순서인가?", "예상 소요 시간", "완료 기준"이 명시되었는가?

#### **🔗 의존성 및 순서**

- [ ] 기술적 의존성이 올바르게 고려되었는가?
- [ ] UI와 백엔드 로직이 적절히 분리되어 독립 개발이 가능한가?
- [ ] 중복 작업을 최소화하는 순서로 배치되었는가?

#### **🧪 테스트 검증**

- [ ] API 연동 및 비즈니스 로직 구현 Task에 Playwright MCP 테스트가 포함되었는가?
- [ ] 각 작업 파일에 "## 테스트 체크리스트" 섹션이 명시되었는가?
- [ ] 모든 사용자 플로우에 대한 E2E 테스트 시나리오가 정의되었는가?
- [ ] 에러 핸들링 및 엣지 케이스 테스트가 고려되었는가?
- [ ] Phase 3에 통합 테스트 Task가 포함되었는가?
